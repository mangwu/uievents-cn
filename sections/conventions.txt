<h2 id="style-conventions">格式规约</h2>


本规范遵循
<a 
    href="http://www.w3.org/People/Schepers/spec-conventions.html">
    建议的W3C规范惯例
</a> ，并添加了以下补充内容: 

*   打印在 <a href="#key-legends"><em>键帽</em></a> 上的键显示为
    KEYCAP{&darr;}， KEYCAP{=} 或 KEYCAP{Q} 。这用于从用户的角度引用键，
    而不考虑生成的 {{KeyboardEvent}} 事件中的 
    {{KeyboardEvent/key}} (键) 和 {{KeyboardEvent/code}} (键码)。

    <p class="note custom-note">
        <q>键帽是放置在计算机键盘的键开关上的塑料、金属或其他材料的小盖子。
        键帽通常用于指示它们所对应的键功能或字母数字字符。</q> —— 
        <a href="https://en.wikipedia.org/wiki/Keycap">英文维基</a>。
        这里使用特定格式来表示键冒上的键是为了避免与 {{KeyboardEvent}} 
        接口上定义的 <code>key</code> 属性的值和 <code>code</code> 属性值区分，
        除此之外也应该与 {{KeyboardEvent}} 和 {{MouseEvent}} 接口上定义的
        <code>ctrlKey</code> ， <code>shiftKey</code> ， <code>altKey</code> ，
        <code>metaKey</code> 属性区分。举一个例子，如果键帽为 KEYCAP{Ctrl} 
        的左控制键被按下，触发一个 EVENT{keydown} 事件，
        那么事件对象的 <code>key</code> 属性值就是 "Control"(KEY{Control})， 
        <code>code</code> 属性值就是 "ControlLeft"(CODE{ControlLeft})， 
        而 <code>ctrlKey</code> 的属性值是 <code>true</code> ，但通过 
        {{KeyboardEvent/getModifierState()}} 方法检查按键按下状态时，
        需要传递的修饰符键值应该是 "Control"(KEY{Control}) 而非 
        <code>ctrlKey</code> 。
    </p>

*   代表字符的字形(Glyphs)显示为: GLYPH{&#x239C2;}。
    
    <p class="note custom-note">
        除了汉字以外，还有很多字形需要使用这种方式表示，
        例如在苹果电脑键盘上的 GLYPH{Command} (GLYPH{&#x2318;}) 键修饰符。
    </p>

*   Unicode字符编码显示为: UNI{U+003d}。

*   按键按下生成的键值名称(即 {{KeyboardEvent}}.{{KeyboardEvent/key}} 的值)
    显示为: KEY{ArrowDown} 、KEY_NOLINK{=} 、 KEY_NOLINK{q} 或 KEY_NOLINK{Q} 。

*   与物理键相关的键代码的名称(即 {{KeyboardEvent}}.{{KeyboardEvent/code}} 
    的值)显示为: CODE{ArrowDown} ， CODE{Equal} 或 CODE{KeyQ} 。


此外，本规范中会使用具有特定含义的一些术语。术语 <q>实现(implementation)</q> 
适用于指代实现本规范的浏览器、内容创作工具或其他 
<a href="#user-agent">用户代理</a> ，而内容作者是指编写脚本或代码的人，
这些脚本或代码利用本规范中描述的接口、方法、属性、
事件和其他功能来制作 Web 应用程序，并且用户是在一个实现中使用这些 
Web 应用程序的人。

<h3 id="ui-events-unofficial-conventions">非官方的翻译规约</h3>

    <strong>本节不是规范的。</strong>

    本节为译者自行添加， <strong>不属于</strong> 英文规范原档内容，
    且本节是对翻译过程中的中文翻译，文本样式以及内容补充说明。

    <h4 id="ui-events-zh-translation">中文翻译</h4>

        本文档翻译的 UI Events 规范基于
        <a href="https://www.w3.org/TR/2022/WD-uievents-20220913/">
        https://www.w3.org/TR/2022/WD-uievents-20220913/</a>。 
        翻译工作量巨大，翻译时借助了辅助工具，难免出现错误或遗漏的地方，
        请不要把本文档当作规范看待，可以当成英文原文档规范的学习参考。

        [[#ui-events-conformance]] 中提到了一些英文大写关键字，
        这些关键字都翻译成了中文，翻译参考自 
        <a href="https://github.com/shibei-rfc/shibei-rfc2119">shibei-rfc2119</a>

    <h4 id="ui-events-zh-text-style">文本样式</h4>

        中文是没有空格的，字符和字符之间挨着，通过标点符号将一句话分隔以便理解。
        但是英文是由单词组成的，每个单词之间通过空格分隔，
        最后再用标点符号进行语义区分。由于英文空格的存在，
        在英文内容源码换行时，
        会自动将新开行的首个单词和上一行的尾部单词之间的回车符号作为空格渲染，
        如果中文文本使用同样的规则，会导致在浏览器渲染的最终结果上，
        汉字之间有不可避免的空格，这不符合阅读习惯。
        为了保证中文的显示效果没有割裂感，
        中文源码内容遵循以下在翻译实践中探寻出的规则: 

        *   标点符号确认—— <br/> 
            冒号，使用英文冒号 <code>: </code> ，并且后带有一个空格; <br/> 
            分号，使用英文分号 <code>; </code> ，并且后带有一个空格; <br/> 
            逗号，使用中文逗号 <code>，</code> ，作为语义的默认分隔符号; <br/> 
            句号，使用中文句号 <code>。</code> ，作为语义的默认结束符号; <br/> 
            顿号，使用中文顿号 <code>、</code> ，作为并列词语的分隔符号; <br/>
            引号，使用英文双引号 <code>"</code> ，作为引用词语的包裹符号; <br/> 
            括号，使用英文括号 <code>()</code> ，作为语义解释的包裹符号; <br/> 
            破折号，使用双横杠 <code>——</code> ，作为语义解释的延续符号; <br/>
        
        *   中文源码内容换行规则: 上述的标点符号中，除了引号外，
            其它标点符号均可以作为换行标记符号，但不允许(源码内容)
            汉字之间出现换行。所以浏览器渲染出来的实际文档内容，有些逗号，
            句号等符号后面会多一个空格，但这不影响中文文本阅读。除此之外，
            在可以出现空格的位置也可以换行，见下面一条规则。

        *   空格符号的使用: 出现链接，斜体，粗体，代码以及在 
            [[#ui-events-zh-translation]] 中描述的关键字等所有特殊样式时，
            需要使用空格将这些词语与文本分隔，一个例子如下 <br/>

            <p class="custom-example" data-content="空格符号的使用">
            我 <span class="zh-upper">必须</span> 这样 <em>排列</em> 才符合 
            [[#ui-events-zh-text-style]] 规定的 <strong>中文格式</strong> 。
            </p>
        
        *   有些英文内容没必要翻译，这个时候使用原本的英文源码内容，
            会出现使用英文逗号，英文单词链接左右没有出现空格的情况，
            请确认这并非错误，例如接口定义时的 IDL 伪代码，
            事件类型中的可信目标对象等。总之，一切规则是为了阅读时更顺畅，
            不影响文档文本的阅读观感。
